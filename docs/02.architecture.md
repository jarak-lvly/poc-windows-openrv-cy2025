# Architecture

## Why use a Linux host?

A Linux Docker host provides direct access to:

* the Linux KVM subsystem
* the `/dev/kvm` virtualization device
* Docker Engine and Docker Compose
* Linux networking devices such as `/dev/net/tun`
* bind-mounted directories for moving files between the Linux host and Windows VM
* scripts that can create, configure, discard, and recreate the Windows environment

The basic [dockur/windows](https://github.com/dockur/windows) Compose configuration passes both `/dev/kvm` and `/dev/net/tun` into the container and adds the `NET_ADMIN` capability required for its virtual networking configuration.

This makes it possible for someone whose primary system is Linux to create a disposable Windows development environment without maintaining a separate physical Windows workstation or a manually configured long-lived Windows VM.

The [dockur/windows](https://github.com/dockur/windows) project specifically lists a Linux host with KVM support as a supported configuration. It also supports certain Windows 11 configurations using Docker Desktop or Podman with nested virtualization enabled, but Windows 10 and macOS Docker Desktop do not provide the required KVM support for this image.

## QEMU and KVM

QEMU provides the virtual hardware required by Windows, including the virtual CPU, memory, disks, network devices, and display devices.

On a Linux host, QEMU can use **KVM**, the Linux kernel’s hardware-virtualization interface. The host’s `/dev/kvm` device is passed into the Docker container:

devices:  
  - `/dev/kvm`

This allows QEMU to use the processor’s Intel VT-x or AMD-V/SVM virtualization features instead of emulating every CPU instruction in software. QEMU documents KVM as a virtualization accelerator, and [dockur/windows](https://github.com/dockur/windows) describes its resulting performance as near-native.

Without KVM, QEMU may fall back to software-based CPU emulation, but that would generally be far too slow for a practical Windows development environment and an OpenRV build.
